# Agentic Building 验收原则 v1

> **在 agentic building 中，生成不是瓶颈，验收才是瓶颈。**
> **所以每个版本必须先定义验收口径，再开始大规模生成。**

## 1. 为什么验收变成了核心功课

当开发代价急剧下降时，稀缺的东西不再是"有人能写"，而变成了**判断什么算写对了**。Agent 天然擅长"生成候选答案"，不天然擅长"判定最优答案"。没有验收标准，agent 会疯狂生产"看起来差不多"的版本，但你根本不知道哪一个才是该留下来的。

这是 agentic building 的一条核心定律：

> 生成变便宜后，判断变昂贵。实现变廉价后，验收变核心。

---

## 2. 三条基本纪律

**纪律一：先定义"什么叫做对"，再让 agent 写。**

每个功能开工前，必须有明确的验收口径。不是"写完看看效果"，而是"先说清楚什么效果才算通过"。这不是官僚流程，而是在 agent 能一晚上给你三版的时代里，防止项目变成流沙的唯一方法。

**纪律二：测试覆盖的是高风险行为，不是表面功能。**

测试最大的价值不是"证明软件完美"，而是**固定系统行为**——当改动发生时，知道哪些行为不应该变；当新 agent 接手时，知道什么不能碰；当半年后回来时，知道版本之间的差异在哪里。测试本质上是在给系统写**行为宪法**。

**纪律三：最终是否合并，由真实使用价值决定，而不是"代码看起来不错"。**

验收不是"点点按钮看看坏没坏"，而是回答：这个功能解决的是不是我当初想解决的问题？这个结果是不是我真的愿意长期使用？这个版本值不值得成为未来版本的父节点？

---

## 3. 验收口径卡

每个功能开工前，子敬必须先补一张验收口径卡。这张卡片是整个开发流程的锚点——CC、Codex、子敬、Litch 所有人都围绕它工作。

| 字段 | 说明 |
|------|------|
| **核心目标** | 这个功能最核心的 1 句话目标 |
| **必须通过的真实用例** | 3 个必须通过的真实使用场景 |
| **不能破坏的旧行为** | 2 个不能被破坏的已有功能 |
| **最容易被"做假"的点** | 1 个最容易被 agent 表面实现但实际没做到的点 |
| **失败时的预期表现** | 出错时应该怎么表现（而不是静默失败） |
| **验收方式** | Litch 最终会用什么方式验收（真实数据 / 特定场景 / 对比测试） |

**示例（V0.8 JSON Import）：**

| 字段 | 内容 |
|------|------|
| 核心目标 | 导入 ChatGPT conversations.json，对话数据进入语义搜索 pipeline |
| 必须通过的真实用例 | ① 小文件导入后能搜到对话内容 ② 同一文件重复导入不产生重复 ③ 修改一段对话后重新导入只更新受影响部分 |
| 不能破坏的旧行为 | ① PDF 导入和搜索不受影响 ② 已有 Topics 不被破坏 |
| 最容易被"做假"的点 | 增量更新——看起来"更新了"但实际是全量重建 |
| 失败时的预期表现 | 返回清晰的错误信息，不留残留数据，临时文件被清理 |
| 验收方式 | 用真实的 conversations.json 在 Docker 环境中完整走一遍导入→搜索→增量更新流程 |

---

## 4. 验收的四个层次

验收不是单一动作，而是分层的判断过程：

| 层次 | 问题 | 谁负责 |
|------|------|--------|
| **行为正确性** | 功能是否按 PRD 描述的方式工作？ | 自动化测试 + 子敬 review |
| **执行真实性** | 代码是否真的做了它声称做的事？（非假增量、非假流式） | Codex 语义 review |
| **产品价值** | 这个功能解决的是不是真实需求？用起来顺不顺？ | Litch 真实场景验收 |
| **系统健康度** | 这次改动有没有让系统变得更难维护？ | Root 外部评审 + 子敬架构 review |

---

## 5. 与其他协议的关系

验收原则是整个团队协作体系的底层哲学，贯穿于所有其他协议之中：

- **[多 Agent 分层 Review 协议 v1](multi-agent-review-protocol.md)**：Review 的终极目的是确保代码达到验收标准。合并标准第 6 条"Litch 验收通过"是最终关卡。
- **[多 Agent Bug 处理协议 v1](multi-agent-bug-protocol.md)**：Bug 的本质是"验收标准被违反"。Bug 关闭时补的"新纪律"会反哺未来的验收口径卡。
- **大需求架构评审流程**：第一步"需求对齐"中的 goals/non-goals，本质上就是在定义验收边界。

---

## 6. 一句话总结

> 每一个版本真正完成的标志，不是"代码写完了"，不是"测试通过了"，不是"review 过了"，而是**在真实场景中，它解决了它承诺要解决的问题**。
