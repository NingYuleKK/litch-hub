# 多 Agent 分层 Review 协议 v1

适用范围：Litch 全部重要工程项目，尤其是涉及导入、增量更新、向量索引、流式处理、事务、部署、权限、迁移的版本。

## 0. 目的

在多 Agent 协作中，把"方向正确、实现真实、问题可追溯、纪律可升级"固化为默认流程，避免出现：
- 设计看起来对，执行其实是假的
- 功能能跑，但未来版本被埋雷
- Bug 修掉了，但没有变成纪律
- 文档缺失，导致后续协作失忆

---

## 1. 核心原则

### 1.1 分层 Review，缺一不可

任何重要版本都至少经过以下三层审查：
1. **架构与版本层 Review**
2. **执行语义层 Review**
3. **真实用户验收层 Review**

### 1.2 找回归点，不找罪人

讨论"谁引入 bug"时，目标是：
- 找到回归点
- 找到最懂上下文的人
- 把 bug 转化为纪律

不是人格归责，不做情绪化追责。

### 1.3 先写护栏，再开工

在开发前必须先明确：
- 本版解决什么
- 本版不解决什么
- 未来 1 到 3 个版本的大方向
- 当前版本的风险点
- 当前版本的架构健康度判断

### 1.4 文档即共同记忆

所有关键协作必须沉淀到文档，不依赖口头同步。最少需要更新：
- memory
- handover
- readme
- issue comment
- bug / decision / discipline 记录

### 1.5 重要改动必须可回滚

凡涉及 schema、导入、索引、部署、权限的改动，必须说明：
- 回滚方式
- 数据是否可恢复
- 失败后系统会留下什么状态

---

## 2. 角色分工

### 2.1 Litch

负责：
- 定义目标、边界、优先级
- 提出真实使用场景
- 最终验收"好不好用"
- 在关键节点裁决版本范围

### 2.2 Root

负责：
- 外部评审
- 风险扫描
- 未来版本兼容性判断
- 架构健康度评估
- 补充护栏与验收口径

### 2.3 子敬 / Manus

负责：
- 写 PRD
- 任务拆分
- 架构与流程守门
- Code Review 的第一层
- 统一汇总反馈，维护项目纪律
- 更新 HANDOVER 与 soul 级规则

### 2.4 CC / Claude Code

负责：
- 根据 PRD 实现代码
- 写测试
- 写 PR
- 修复 review 反馈
- 保证提交是可运行、可验证的

### 2.5 Codex

负责：
- 执行语义 Review
- 检查"说一套做一套"的风险
- 重点盯：
  - 假增量
  - 假流式
  - 半写入 / 事务不闭合
  - 幂等性与竞态
  - 测试覆盖是否命中高风险路径

---

## 3. 分层 Review 结构

### Layer 0：目标与护栏评估

负责人：Litch + Root + 子敬

开发前必须回答：
1. 这版解决什么需求？
2. 这版明确不解决什么？
3. 未来 1 到 3 个版本大概怎么演化？
4. 本版最大的 3 到 5 个风险是什么？
5. 架构健康度判断是什么？
6. 验收口径是什么？
7. 回滚方案是什么？

### Layer 1：架构与版本层 Review

负责人：子敬

重点看：
- schema 是否干净
- 模块边界是否合理
- migration 是否完整
- 文档是否齐
- 配置、部署、依赖是否可持续
- 未来扩展有没有被堵死

### Layer 2：执行语义层 Review

负责人：Codex

重点看：
- 代码是否真的实现了 PRD 承诺
- 数据流是否真实闭合
- 流式、增量、事务是否只是表面存在
- 状态机是否一致
- 高风险路径是否被测试覆盖

### Layer 3：真实用户验收层 Review

负责人：Litch

重点看：
- 能不能跑
- 好不好用
- 是否符合真实场景
- 性能是否可接受
- 错误是否可理解
- 文案和操作是否足够直觉

---

## 4. 哪些改动必须触发 Codex Review

以下类型默认强制进入执行语义 Review：
- 导入类功能
- 增量更新
- chunk / embedding / index 重建逻辑
- 流式处理
- 事务、幂等、恢复机制
- schema migration
- auth / 权限 / 密钥管理
- Docker / 部署模式切换
- 删除旧数据并重建的逻辑
- 一切"看起来像优化，实际上可能是假优化"的改动

---

## 5. 每次版本必须具备的交付物

### 5.1 Issue / PRD 必须包含

- 版本号
- 目标
- 非目标
- 风险列表
- 架构健康度判断
- 验收口径卡
- 回滚方式
- 相关 sample / fixture

### 5.2 PR 必须包含

- 改动摘要
- 涉及文件列表
- migration 说明
- 测试说明
- 已知限制
- 部署影响
- 文档更新说明

### 5.3 Review 结论必须包含

- 问题等级：P0 / P1 / P2
- 哪个问题必须本次修
- 哪个问题可延期
- 为什么
- 由此新增的纪律是什么

---

## 6. Bug 处理协议

### 6.1 发现 bug 后必须记录

- 现象
- 复现路径
- 回归点
- 根因
- 修复方式
- 新增测试
- 新纪律

### 6.2 Bug 记录的目标

不是为了说明"谁错了"，而是为了沉淀：
- 哪个环节的纪律失效了
- 以后如何避免
- 哪类 PR 以后必须多一层 review

---

## 7. 合并标准

一个版本只有在以下条件同时满足时才允许合并：
1. PRD 范围没有失控
2. 架构 Review 通过
3. 执行语义 Review 通过
4. 自动化测试通过
5. Docker / 本地运行通过
6. Litch 验收通过
7. handover / readme / memory / discipline 文档已更新

---

## 8. 版本结束后必须做的事

每个版本完成后，至少更新以下内容：
- `HANDOVER`
- `README`
- `memory`
- `discipline` 或 `soul.md`
- 关键 `decision log`
- 关键 `bug log`

并补一句：
- 这一版最重要的设计决策是什么
- 这一版新增的纪律是什么

---

## 9. 本协议的核心精神

多 Agent 协作的关键，不是谁最强，而是：
- 方向有人守
- 架构有人守
- 执行真假有人守
- 用户价值有人守
- 所有经验最后都能写回纪律

只有这样，系统才会越做越稳，而不是越做越乱。
