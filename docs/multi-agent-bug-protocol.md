# 多 Agent Bug 处理协议 v1

> **先分类，再定 owner；先查根因，再落 patch；一 bug 一写手。**

## 1. 背景与目标

当系统进入深水区（Docker 环境、数据导入、增量状态、流式处理等多层交织），Bug 往往不再是"拼错一行代码"，而是**多层接口之间没有真正闭合**。此时最容易乱掉的不是技术本身，而是"谁来查、谁来改、谁来拍板"三件事缠在一起。

本协议的目标是：**让每一个 Bug 在任一时刻只有一个明确的 Owner，从发现到关闭形成完整闭环。**

---

## 2. 四角色分工

### Litch — 验收者 + 优先级裁决者

Litch 只负责三件事：说清楚现象、说清楚预期、判断修完后是不是"真的好用"。Litch 不下场做详细排查协调，避免被切碎。

### 子敬 — Bug Triage / 项目经理

子敬负责收集 Bug 信息、给 Bug 分类、指定本次谁查谁写谁 review、追踪 Bug 最后是否变成纪律。子敬不一定亲自查每一行代码，但必须掌握 8 个关键信息（见第 5 节）。子敬最像**事故总指挥**——不是亲自下井修机器，而是确保大家都在看同一个问题、有明确 Owner、修完后有文档和纪律沉淀。

### Codex — 排查者 / 执行语义侦探

Codex 最适合查"为什么会这样"——runtime、Docker、流式、增量、事务、状态不一致，以及"代码说自己做了但其实没做到"的问题。Codex 像法医，不一定总是主刀医生。

### Claude Code (CC) — 主写手 / 修复执行者

CC 最适合按明确诊断去改代码、补测试、提 PR、把 fix 落地成可合并的东西。**默认组合：Codex 查，CC 改。**

---

## 3. 默认流程

### Step 1：Litch 提交 Bug（只提现象，不提解决方案）

提交给子敬的信息：

| 字段 | 说明 |
|------|------|
| 复现步骤 | 具体操作序列 |
| 实际结果 | 截图 / 日志 / Docker 输出 |
| 预期结果 | 应该是什么样 |
| 版本信息 | commit / image tag / 分支 |
| 测试数据 | 用的哪个 sample / conversation.json |

**重要：不要一上来猜根因（如"应该是 embedding 的问题"），避免污染排查方向。**

### Step 2：子敬做 Triage，判断 Bug 类型

| 类型 | 特征 | 处理方式 |
|------|------|---------|
| **A. 验收偏差 / 功能不符合预期** | 导入后 topic 不对、UI 表现不对、覆盖逻辑和预期不一样 | 先看 PRD，判断是实现错了还是需求没写清 → CC 改，Codex review |
| **B. 执行语义 Bug / 深层逻辑 Bug** | 假增量、假流式、半写入、Docker 下状态失真 | Codex 先做 root cause analysis → 出报告 → CC 改 |
| **C. 环境 / Docker / 配置 Bug** | 容器跑不起来、volume 映射问题、环境变量不生效 | Codex 先查运行链路 → 谁最适合写 fix 谁写，但只保留一个主写手 |
| **D. PRD 缺口 / 决策未定** | conversation 覆盖是 merge 还是 replace 等产品决策 | 先不查代码 → 子敬找 Litch 做产品裁决 → 定完规则再继续 |

### Step 3：排查（如需要）

若根因不明确，由 Codex 进行 root cause analysis，输出一份简短的根因报告，包含：问题定位、影响范围、建议修复方向。

### Step 4：修复

由指定的主写手（通常是 CC）执行修复，包括改代码和补测试。

### Step 5：子敬 Review

验证修复是否匹配根因与 PRD，确认没有引入新问题。

### Step 6：Litch 验收

Litch 在真实环境中确认"真的好用了"。

### Step 7：关闭 Bug，补齐闭环

Bug 关闭前，必须补齐以下四项：

| 项目 | 说明 |
|------|------|
| **Root Cause** | 一句话总结根因 |
| **回归点** | 这个 Bug 是从哪个 commit / 哪次改动引入的 |
| **新纪律** | 转化为一条防止复发的规则 |
| **新测试** | 补充的回归测试用例 |

---

## 4. 核心纪律

### 一 Bug 一写手

> **排查可以多人，落 patch 只能一人。**

以下做法是允许的：
- Codex 查，CC 改
- CC 查，CC 改
- Codex 查，Codex 改

以下做法是**禁止的**：
- Codex 和 CC 同时各写一版 fix
- 子敬一边 review 一边偷偷自己补逻辑
- Litch 在测试时又让另一个 agent 顺手改一点

违反这条纪律会导致：根因没对齐、分支冲突、谁修好了说不清、纪律写不回去。

---

## 5. 子敬必须掌握的 8 个信息

对于每一个 Bug，子敬作为 Triage 负责人必须掌握：

| # | 信息 | 说明 |
|---|------|------|
| 1 | 版本 | 哪个 commit / image tag |
| 2 | 数据 | 用什么 sample / conversation.json 复现 |
| 3 | 复现步骤 | 操作序列 |
| 4 | 预期 vs 实际 | 差异描述 |
| 5 | Bug 分类 | A/B/C/D 哪一类 |
| 6 | 当前 Owner | 谁在查、谁在改 |
| 7 | 根因总结 | 一句话 |
| 8 | 闭环状态 | 补了什么测试、什么纪律 |

子敬**不需要**亲自追每个 stack trace、自己写最终 patch、看所有细节日志。

---

## 6. 与 Review 协议的关系

Bug 处理协议与[多 Agent 分层 Review 协议 v1](multi-agent-review-protocol.md)互补：

- Review 协议管的是**合并前的质量关**：先护栏，后开工；先分层 review，再谈合并
- Bug 协议管的是**合并后的问题闭环**：先分类，再定 owner；先查根因，再落 patch

两者共同确保：**每一个问题都有明确的 Owner、每一个修复都有对应的纪律沉淀。**

- [Agentic Building 验收原则 v1](agentic-building-acceptance-principles.md)：验收是整个协作体系的底层哲学

---

## 7. 快速口令

团队日常沟通中，可以用以下口令快速对齐：

- **"先分类"** → 子敬判断 A/B/C/D
- **"定 owner"** → 明确谁查谁改
- **"查根因"** → Codex 出 root cause 报告
- **"落 patch"** → CC 写代码补测试
- **"补纪律"** → 关闭前写 root cause + 回归点 + 新纪律 + 新测试
