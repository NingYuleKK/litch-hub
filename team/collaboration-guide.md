# 协作模式说明

> 每次新任务，由 Litch 决定走哪条路线。

---

## 团队两轴结构

团队分为**思考轴**和**工作轴**两条协作主线：

```
思考轴（Chat LLM）           工作轴（Agent / Claw）
─────────────────           ─────────────────────
Litch（最终决策）    ──→     子敬（Manus）项目管理 & 部署
Root（GPT）战略讨论           奉孝（OpenClaw 🦞）调研
小克（Claude）对比分析         元直（KimiClaw）执行 & 研究
                              Codex 代码编写
```

**信息流向：** 思考轴产出决策 → 子敬接收并拆解任务 → 工作轴各成员分工执行 → 子敬验收 → Litch 确认。

---

## 两条执行路线

### 路线 A：子敬写 Issue Spec → Codex 写代码 → 子敬验收部署

**适用场景：** 功能较复杂、需要精确规格说明、或希望 Codex 独立完成大块代码的任务。

**流程：**

```
Litch 确定需求（思考轴讨论后拍板）
    ↓
子敬将需求拆解为 GitHub Issue spec
    ↓
Codex 根据 Issue spec 编写代码
    ↓
子敬验收代码（功能测试 + 代码审查）
    ↓
子敬部署上线
    ↓
Litch 最终确认
```

**优点：**
- 规格清晰，Codex 实现质量更高
- 子敬可以并行处理其他任务
- 代码有 spec 文档，便于后续维护

**缺点：**
- 流程较长，来回沟通成本较高
- 适合功能明确、边界清晰的任务

---

### 路线 B：子敬直接写代码 + 部署

**适用场景：** 功能较简单、需要快速迭代、或子敬对项目已有充分上下文的任务。

**流程：**

```
Litch 确定需求
    ↓
子敬直接编写代码
    ↓
子敬部署上线
    ↓
Litch 最终确认
```

**优点：**
- 速度快，适合小功能和 hotfix
- 减少沟通层级，直接产出结果

**缺点：**
- 子敬的代码能力不如 Codex，复杂功能可能质量较低
- 缺少 spec 文档，后续维护成本略高

---

## 路线选择参考

| 情况 | 推荐路线 |
|------|---------|
| 新功能，逻辑复杂 | 路线 A |
| 小改动、样式调整 | 路线 B |
| Bug 修复 | 路线 B |
| 全新页面或模块 | 路线 A |
| 紧急上线 | 路线 B |
| 需要高质量代码 | 路线 A |

---

## 调研支撑流程

当工作轴需要调研支撑时（如技术选型、竞品分析），子敬可向奉孝（OpenClaw）或元直（KimiClaw）发起调研请求：

```
子敬 识别调研需求
    ↓
奉孝 / 元直 执行调研，输出报告
    ↓
子敬 整合调研结论，推进工程决策
    ↓
重要结论归档至 litch-hub/research/
```

---

## 任务启动模板

每次新任务开始时，Litch 请提供：

1. **任务描述**：要做什么？
2. **路线选择**：走路线 A 还是路线 B？
3. **目标仓库**：哪个项目？
4. **验收标准**：怎样算完成？

---

## 知识库更新规范

当有新的调研报告、IP 设定或项目文档产生时，相关 Agent 应及时更新本仓库（`litch-hub`），确保团队知识库保持最新状态。

| 目录 | 维护责任人 |
|------|-----------|
| `projects/` | 子敬 |
| `team/` | 子敬 |
| `research/` | 奉孝 / 元直 |
| `ip/` | Litch（或委托子敬整理） |
