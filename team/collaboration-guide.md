# 协作模式说明

> 每次新任务，由 Litch 决定走哪条路线。

---

## 两条协作路线

### 路线 A：子敬写 Issue Spec → Codex 写代码 → 子敬验收部署

**适用场景：** 功能较复杂、需要精确规格说明、或希望 Codex 独立完成大块代码的任务。

**流程：**

```
Litch 确定需求
    ↓
子敬（Manus）将需求拆解为 GitHub Issue spec
    ↓
Codex 根据 Issue spec 编写代码
    ↓
子敬验收代码（功能测试 + 代码审查）
    ↓
子敬部署上线
    ↓
Litch 最终确认
```

**优点：**
- 规格清晰，Codex 实现质量更高
- 子敬可以并行处理其他任务
- 代码有 spec 文档，便于后续维护

**缺点：**
- 流程较长，来回沟通成本较高
- 适合功能明确、边界清晰的任务

---

### 路线 B：子敬直接写代码 + 部署

**适用场景：** 功能较简单、需要快速迭代、或子敬对项目已有充分上下文的任务。

**流程：**

```
Litch 确定需求
    ↓
子敬（Manus）直接编写代码
    ↓
子敬部署上线
    ↓
Litch 最终确认
```

**优点：**
- 速度快，适合小功能和 hotfix
- 减少沟通层级，直接产出结果

**缺点：**
- 子敬的代码能力不如 Codex，复杂功能可能质量较低
- 缺少 spec 文档，后续维护成本略高

---

## 路线选择参考

| 情况 | 推荐路线 |
|------|---------|
| 新功能，逻辑复杂 | 路线 A |
| 小改动、样式调整 | 路线 B |
| Bug 修复 | 路线 B |
| 全新页面或模块 | 路线 A |
| 紧急上线 | 路线 B |
| 需要高质量代码 | 路线 A |

---

## 任务启动模板

每次新任务开始时，Litch 请提供：

1. **任务描述**：要做什么？
2. **路线选择**：走路线 A 还是路线 B？
3. **目标仓库**：哪个项目？
4. **验收标准**：怎样算完成？

---

## 知识库更新规范

当有新的调研报告、IP 设定或项目文档产生时，相关 Agent 应及时更新本仓库（`litch-hub`），确保团队知识库保持最新状态。

- **子敬**负责维护 `projects/` 和 `team/` 目录
- **元直**负责维护 `research/` 目录
- **Litch**负责维护 `ip/` 目录（或委托子敬代为整理）
