## 影子策略（Shadow Strategy）

在不打扰主线路、不做大规模动员的前提下，针对某个趋势或风险，先行搭建的小规模、低可见度、可随时撤退的试验性策略路径。

**特征：**
- 与"公开主战略"并行存在，但不要求立刻对团队或市场做承诺
- 资源投入可控：时间、资金、人力都设有上限，可以随时关停或转向
- 关注的是"验证假设 / 把坑踩一遍"，而不是 KPI 级别的结果
- 适合用于：新模型、新平台、新业务形态的探索

**在 Litch 语境下的常见场景：**
- 在没有对公司正式宣布前，先用个人 / 小团队方式测试某种 Agent 工作流
- 在对外产品还没做大改变前，先用内部工具试水新的交互范式
- 针对潜在的政策 / 行业变化，预先搭建"如果 X 发生，Y 方案可以立刻上线"的线路

**目的：** 让 Litch 可以同时保持对未来的敏感与提前布局，以及对当前团队与自身带宽的责任感和可控性。

## 对话资产（Conversation Assets）

所有由 Litch 与人类 / Agent 之间发生的可记录对话及其衍生物，在被妥善整理、索引、结构化之后，成为可以反复查询、复用、再创作的"长期生产资料"。

**包含的内容类型：**
- Chat 记录（ChatGPT / OpenClaw / Manus / Kimi 等）
- 语音转写文本、IM / 邮件中的关键讨论
- 对话衍生的：思维导图、spec、PRD、项目文档、blog 草稿
- 与对话强相关的：引文、截图、评论、批注

**从"聊天记录"到"对话资产"的关键变化：**
- **有落点**：进入 litch-hub 或对应的知识库，而不是散落在各个 App
- **有结构**：最少有时间、主题、项目归属、关联人的标记
- **可复用**：未来的 Agent / 人类可以基于这些资产继续推演，不必从零开始

**在系统中的落点：**
- `reflection/`：带批注与元分析的对话片段
- `projects/`：对话资产被抽取为 spec / 决策记录
- `ip/`：成为长期写作与创作工程的素材库

**核心理念：** 对话不是一次性消耗品，而是"认知与工程共同生产的原材料"，通过治理与结构化，它们可以变成 Litch 一生都在使用的知识底座。

## 家族看板（Family Board）

围绕 Litch 及其核心 Agents 家族构建的、面向"目标与协作"的统一视图集合。不是某个具体工具，而是一整套：文件结构 + 目标设计 + 节奏约定。

**组成：**
- **北极星层**：`goals/litch-okr-2026H1.md` 等，核心 Objective 与 KR
- **当下层**：`goals/now.md`，当前周期的三件大事、不做清单、KR 追踪
- **执行层**：`goals/agents/*.md`（各 Agent 执行看板）+ `goals/kr-map.md`（KR 映射表）
- **记忆与决策层**：`logs/decisions.md`、节奏记录、系统回顾等

**功能：**
- 为 Litch 提供"一眼能看出现在全家在干什么"的总览
- 让每个 Agent 明确知道：自己所属的目标、边界与交付物
- 承接 Project Conductor 的组织设计

**原则：**
- "事实在文件，视图在工具"：家族看板本质上是 repo 中的一组文件结构，任何可视化看板都只是镜像
- 优先记录"决策当时的上下文"，而不仅是结果
- 保持轻量：宁可信息略少，也不要成为另一种沉重的行政负担

## Agentic Coding / Agentic Building

不是"随手写写代码玩玩"，而是以「人与 Agent 协作」为核心前提，有意识地设计、构建并迭代工作流、工具链、权限结构和系统组件的实践方式。

**与 vibe coding 的区别：**

| | vibe coding | agentic coding / building |
|--|------------|--------------------------|
| 入口 | 情绪：好玩、上头、临时灵感 | 结构：从"我要为谁打造什么能力"开始 |
| 目标 | 模糊，更多是探索试验 | 明确，例如"为 Project Conductor 增加一条可复用流水线" |
| 产出 | 不一定可复用 | 强调可交接性与可持续性 |

**典型内容：**
- 为 Agent 设计记忆写入 / 读取机制
- 构建项目工作流（对话 → spec → 代码 → 部署 → 回顾）
- 设计权限边界（谁能动本地、谁只能读 repo）
- 构建"人类 × Agent × 工具"的闭环

**为什么要用这个词：** 它提醒 Litch：自己不只是"在玩工具"，而是在建设一种新的组织与生产方式。与 Agent 的协作不是临时 buff，而是需要被设计和治理的工程行为。

## Agent

Litch 团队中承担具体执行职能的 AI 实体。区别于纯对话的 LLM（如 Root / 小克），Agent 具备操作工具、读写文件、执行命令的能力。
> 例：奉孝、子敬、元直、Codex

---

## HANDOVER.md

每个项目必须有的交接文档。格式：项目概述 + 技术架构 + 当前进展 + 启动方式 + 已知问题。目的：奉孝 session 重启后 5 分钟内恢复上下文。

---

## Single Source of Truth（唯一事实来源）

litch-hub 仓库的定位。所有 Agent 以此为准，不以 session 记忆、本地文件或口头约定为准。

Litch 的个人行为准则：不傲慢、不偏见、永远真诚、永远愚者、永远 grokking。出现在 Litch 画像的红线部分。

Litch 对 AI / 算力 / 经济 / 社会演化的整体观察框架，视之为一个开放场域。出自《带宽民族志》主线。
